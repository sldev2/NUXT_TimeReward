# Nuance for Deleting package-lock.json

## The Rule

**Never delete `package-lock.json` and run `npm install` expecting to get the same dependency tree back.** You won't -- npm will resolve the latest versions within every semver range, including for transitive dependencies you don't control.

## When to Keep the Lockfile (Default)

- **Day-to-day development** -- use `npm ci` for clean installs
- **CI/CD pipelines** -- always `npm ci`
- **Onboarding new developers** -- clone + `npm ci`
- **After a crash or corrupted `node_modules`** -- delete `node_modules`, keep the lockfile, run `npm ci`
- **After a failed debugging session** -- if someone accidentally deleted the lockfile, restore it from git (`git restore package-lock.json`) before installing

## When Deleting the Lockfile Is Acceptable

- **Intentional full upgrade** -- you want to pull the latest allowed versions of all dependencies. Delete the lockfile, run `npm install`, test thoroughly, and commit the new lockfile.
- **Resolving deeply tangled dependency conflicts** -- when version conflicts can't be resolved by `npm install <package>` alone and the lockfile has become internally inconsistent.

In both cases:
1. Work in a git branch
2. Test login, core features, and cross-browser sync before merging
3. Commit the new `package.json` AND `package-lock.json` together

## What `npm ci` vs `npm install` Actually Does

| Command | Reads lockfile? | Updates lockfile? | Use case |
|---------|----------------|-------------------|----------|
| `npm ci` | Yes -- installs exact versions | No -- fails if lockfile is out of sync with `package.json` | Clean installs, CI/CD, reproducible builds |
| `npm install` | Yes, but resolves newer versions within semver ranges | Yes -- rewrites lockfile with new resolutions | Adding/upgrading packages intentionally |
| `npm install <pkg>` | Partially -- only updates the target package tree | Yes -- updates only the affected portion | Adding or upgrading a single dependency |

## What Happened in This Project (Feb 26, 2026)

The lockfile was deleted during a debugging session, and `npm install` was run. This pulled newer versions of `@supabase/gotrue-js` (a transitive dependency inside `@supabase/supabase-js`) that had a `navigator.locks` deadlock bug ([supabase-js#2013](https://github.com/supabase/supabase-js/issues/2013)). The login page rendered, but the Supabase auth client deadlocked during initialization, preventing all authentication operations.

The fix was trivial once diagnosed:
```
git restore package.json package-lock.json
npm ci
```

This restored the exact working dependency tree from git and reinstalled it. Hours of debugging a `navigator.locks` patch were unnecessary -- the real solution was just restoring the lockfile.

## Key Takeaway

The lockfile is the source of truth for reproducible builds. Treat it like production code: commit it, review changes to it, and don't delete it casually. When you do delete it intentionally, treat it as a deliberate upgrade event with testing before merge.
