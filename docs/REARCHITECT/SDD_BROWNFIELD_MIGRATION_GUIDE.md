# High-Level Guide for AI Agents: Evolving Brownfield Projects with Spec-Driven Development (SDD) via intent-driven-template

**Intended Audience**: AI coding agents (Claude in Cursor, Windsurf/Cascade, OpenCode, etc.) assisting a solo founder with an existing SaaS codebase (Nuxt/Vercel/Supabase stack or similar).  
**Purpose**: Provide durable, referenceable instructions so the agent consistently applies best-practice SDD to brownfield work instead of ad-hoc changes or risky full reverse-engineering.  
**Core Source**: Based on the official `intent-driven-template` (https://github.com/intent-driven-dev/intent-driven-template) + the Spec-Driven Development brownfield methodology by Hari Krishnan / Intent Driven Dev (OpenSpec + incremental approach).  
**When to Refer to This Guide**: Any time the user asks to add a feature, fix a bug, refactor, or "migrate" / improve an existing project that lacks comprehensive upfront specifications.

---

## 1. Definition: What Makes a Project "Brownfield" in SDD?

A **brownfield project** (in the Spec-Driven Development sense) is any codebase that lacks a reliable, human- or LLM-authored source-of-truth specification set that drove its original creation.

- Analogy (Michael Feathers, *Working Effectively with Legacy Code*): Code without tests = legacy code.  
- Parallel: Code without specs (even if recently generated by Bolt/Replit/Cursor/etc.) = brownfield for SDD purposes.

**Greenfield** = project built from clear, explicit specifications from the start.  
**Goal of migration**: Incrementally move high-change areas toward greenfield (rich, trustworthy specs) while leaving stable/dead code as brownfield. Do **not** attempt a one-time full reverse-engineering of the entire codebase.

---

## 2. Core Principle: Incremental Focused Specs > Full Reverse-Engineering

**Never** attempt to reverse-engineer a complete set of specifications from the entire existing codebase upfront.

**Why it fails**:
- High review burden and token cost.
- Derivative artifacts → information loss, plausible but incorrect assumptions about original intent.
- When later building new features against these imperfect specs, you introduce regressions and drift from the real source-of-truth (the running code + data model).
- Example from Run Club app case study: Reverse-engineered spec for removing "Maybe" RSVP missed a critical DB check constraint and the need for data migration. Incremental approach (starting fresh on the change) immediately surfaced both.

**Recommended Approach (Analogous to Characterization Tests in TDD)**:
- For **each new change** (feature addition, modification, or removal), author **focused, narrow-scope specifications directly from the current codebase**.
- Use the codebase itself as the immutable source of truth for that change.
- Over time, frequently touched areas accumulate high-quality specs and become effectively greenfield. Stable areas can remain brownfield.
- This is far more economical in time, tokens, and accuracy.

**Key Mantra for the Agent**: "Explore the relevant slice of code for *this specific change*, propose precisely, then implement. Park unrelated cleanup for later."

## 3. Recommended Workflow for Any Change in a Brownfield Project

Follow the `intent-driven` schema workflow (proposal → specs → design → ADR → tasks) **incrementally**, with strong emphasis on the Explore phase.

### Phase 0: Setup (One-Time per Project)
1. Ensure the `intent-driven-template` has been installed into the target project (see Section 4).
2. Confirm `openspec/config.yaml` uses `schema: intent-driven`.
3. Configure Repomix MCP (highly recommended — see Section 6).

### Phase 1: Explore (Mandatory for Brownfield)
- **Always start in Explore mode** (`/opsx:explore` or equivalent OpenSpec explore skill, or custom prompt).
- Goal: Learn the relevant parts of the codebase + clarify scope/edge cases with the user.
- **Critical Tool**: Use **Repomix MCP** (or CLI) to pack *only the relevant directories/files* for the proposed change. Do not dump the entire monorepo if it exceeds context.
  - Example instruction: "Use Repomix MCP with include patterns for `src/components/rsvp/`, `src/lib/db/`, and the migration folder. Ignore node_modules, .next, etc."
- During Explore, ask clarifying questions (e.g., "What should happen to existing 'Maybe' RSVPs? Migrate to 'No'?").
- Produce helpful diagrams (frontend/backend/DB changes) if the template's C4 skill is available.
- Output: Clear understanding + parked unrelated issues (e.g., "code duplication noted for later").

### Phase 2: Propose & Author Artifacts Incrementally
- Use `/opsx:continue` (or `openspec-propose`, `openspec-continue-change`, etc.) to move through:
  1. **Proposal** — Why this change? Business/technical intent. Flag if it is a *modification* (not pure addition/removal).
  2. **Specs** — Observable behavior via Gherkin scenarios (use `gherkin-authoring` skill). Keep narrow.
  3. **Design** — How it will be implemented (C4 diagrams via `c4-diagrams` skill, trade-offs).
  4. **ADR** (if architectural impact) — via `architectural-decision-records` skill. Record rationale, supersession.
  5. **Tasks** — Actionable implementation items derived from the above.
- **Adversarial Authoring** (template strength): Use author + reviewer agent pair (`adversarial-authoring` skill) to reduce model bias/hallucination.
- **Grill-Me**: After drafting design/proposal, run `grill-me` interrogation to surface hidden assumptions.
- **Glossary**: Maintain domain terms consistently (`glossary` skill).

### Phase 3: Review, Accept, Apply, Implement
- User reviews artifacts one-by-one.
- Merge proposal to `main` (enforce git discipline — see Section 7).
- Apply change (creates implementation branch/worktree).
- Implement tasks while referencing the accepted specs/design.
- Verify (tests, `openspec-verify-change`).
- Archive after merge to main.

### Phase 4: Post-Change
- Run affected tests (characterization + new).
- If data model changed: Explicitly include migration scripts + backfill logic in the proposal/spec.
- Update glossary or ADRs if new terms/decisions emerge.
- Celebrate incremental greenfield growth in that module.

## 4. Installing the intent-driven-template into an Existing (Brownfield) Project

Follow `INSTALL_TEMPLATE.md` from the template repo (do **not** blindly copy).

**High-Level Steps for the Agent**:
1. Clone the template repo temporarily (or user provides it).
2. Inspect target project for existing `openspec/`, `.opencode/`, `.agents/`, `opencode.json`, `AGENTS.md`, `README.md`.
3. **Copy** (only if missing): `openspec/`, `.opencode/`, `.agents/`, `skills-lock.json`, `opencode.json`.
4. **Merge** (append, do not overwrite):
   - `AGENTS.md` — add the OpenSpec git-discipline instructions.
   - `README.md` — only if user explicitly approves.
5. Validate: `openspec schema validate intent-driven` (if OpenSpec CLI available).
6. Summarize exactly what was added/changed via `git status --short`.
7. Never touch product code, package.json, existing tests, or user docs without explicit request.

After install, the project gains:
- `openspec/` with `intent-driven` schema (local copy for full lifecycle).
- Rich skill set for OpenCode / compatible agents.
- Enforced workflow and git discipline.

## 5. Special Configuration & MCP Server Recommendations for intent-driven-template Development

The template itself is **OpenCode + OpenSpec** centric, but works excellently alongside Cursor + Claude or Windsurf + Cascade when the right MCP servers are available.

### Must-Have MCP for Brownfield Work
**Repomix MCP Server** (official support in Repomix):
- **Why critical**: Allows the AI to intelligently pack *slices* of the brownfield codebase (with smart ignore patterns, token optimization, structure preservation) directly into context during Explore/Continue phases. Essential for staying grounded in the real source of truth without manual `repomix` CLI runs every time.
- **Setup** (user or agent can guide):
  - Install globally or use npx: `repomix --mcp`
  - In **Cursor**: Settings → MCP → Add global MCP server (example config uses `npx -y repomix --mcp` or local path to built server).
  - In **Windsurf/Cascade**: Add via MCP server configuration (similar to existing Vercel/Stripe/Supabase/GitHub/PowerShell MCPs).
  - In **Claude Desktop / Code**: Use `claude mcp add` or edit `claude_desktop_config.json`.
- **Usage Pattern in Prompts**:
  > "For this brownfield change to the RSVP feature, first use the Repomix MCP tool to pack only the relevant frontend components, API routes, database schema/migrations, and Supabase client code. Then explore..."

- Alternative (fallback): CLI `repomix` with `--include` / `--ignore` flags + paste output, but MCP is far superior for agentic flow.

### Other Beneficial MCPs / Tools (Already in User's Stack)
- **GitHub MCP** — for repo operations, PRs, branch management (aligns with git discipline).
- **Supabase MCP** (if using) — for schema inspection, migration planning.
- **Vercel MCP** — deployment context.
- **PowerShell / Filesystem MCPs** — for local env, scripts, .env handling.
- **OpenCode** (if installed) — provides native skills (`openspec-explore`, `adversarial-authoring`, `grill-me`, `c4-diagrams`, `gherkin-authoring`, `glossary`, `openspec-git-discipline`, bulk apply in worktrees, etc.).

### Template-Specific Config Tips
- `openspec/config.yaml`: Ensure `schema: intent-driven` (enables full proposal→spec→design→ADR→tasks lifecycle + git discipline).
- `.agents/skills/` and `.opencode/skills/`: These contain reusable, version-locked skills. Prefer invoking them (e.g., "run adversarial-authoring on this proposal") rather than raw generation.
- `skills-lock.json`: Keeps reproducibility across agent runs.
- For Windsurf "get-shit-done" (GSD) framework users: The template's structured artifacts pair well with GSD loops; use OpenSpec tasks as the "Apply" phase input.

**Recommended Project `.cursor/rules` or Windsurf rules addition** (agent can suggest):
> "When working on this brownfield project, ALWAYS follow the SDD_BROWNFIELD_MIGRATION_GUIDE.md. Start every non-trivial change with Explore + Repomix MCP on the relevant slice only. Use incremental focused specs. Never attempt full codebase reverse-engineering."

## 6. Git Discipline (Enforced by Template)

The `intent-driven` schema + `openspec-git-discipline` skill enforces:
- Proposals must land on `main` **before** any implementation branch is created/applied.
- Implementation must merge to `main` **before** the change can be archived.
- This prevents drift and ensures specs stay authoritative.

Agent should remind user / block if this is violated.

## 7. Best Practices & Common Pitfalls

**Do**:
- Keep every spec/proposal **narrow and change-focused**.
- Explicitly call out data migrations, check constraints, backfills, and breaking changes in the proposal.
- Use characterization-style thinking: "What does the current code actually do here?" before defining desired new behavior.
- Leverage adversarial authoring + grill-me for quality.
- Maintain a living Glossary.
- Let spec coverage grow organically in hot paths.
- Park scope creep ("we'll handle duplication in a future change").

**Don't**:
- Reverse-engineer the whole app "just in case".
- Generate specs in a vacuum without Repomix/context from real code.
- Skip the Explore phase on brownfield changes.
- Accept AI-generated specs without user review of key edge cases (especially data).
- Mix unrelated refactors into the current change's proposal.

**Token / Context Management**:
- Always prefer Repomix MCP with tight include patterns over full-repo dumps.
- For very large codebases, explore one bounded context or module at a time.

---

## 8. Long-Term Outcome

After consistently applying this guide:
- High-activity modules (e.g., auth, payments, core features) become well-specified and pleasant to evolve (greenfield-like).
- Stable or legacy modules remain untouched until a real change is needed.
- The project gains traceability, reduced regression risk, better onboarding, and AI-agent leverage.
- You avoid the common brownfield trap of "we don't know what this does anymore."
